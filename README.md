# Vido Wallpaper Numbers

一个用于批量整理 `.mp4` 视频文件的 Windows 小工具。

## 快速使用

1. 将 `VidoWallpaperNumbers.exe` 放到需要整理的视频目录。
2. 确保该目录第一层中有需要处理的 `.mp4` 文件。
3. 双击运行 `VidoWallpaperNumbers.exe`。
4. 在输入框中填写“每个子文件夹放多少个视频”，允许范围 `1-1000`，默认值 `30`。
5. 点击“开始”。
6. 等待进度条完成。

程序关闭时，会把窗口位置、窗口大小和分组数量保存到 `exe` 同目录下的 `VidoWallpaperNumbers.config.json`。下次启动时会自动恢复。

## 当前版本摘要

- 文件名格式：`四位编号_当天日期.mp4`
- 子文件夹名格式：`起始编号-结束编号_当天日期`
- 处理范围：只处理 `exe` 所在目录第一层的 `.mp4`
- 排序规则：按修改时间从早到晚；修改时间相同则按文件名排序
- GUI：暗色模式窗口，只包含数字输入框、开始按钮、进度条
- 配置保存：关闭窗口时保存窗口位置、窗口大小、分组数量
- 响应性：文件处理在后台执行，GUI 消息循环固定在同一 OS 线程
- 诊断日志：代码保留，但默认关闭，不主动输出日志文件

示例：

```text
0001_20260310.mp4
0002_20260310.mp4
0001-0030_20260310
VidoWallpaperNumbers.config.json
```

## 功能说明

程序会处理 `exe` 所在目录第一层中的所有 `.mp4` 文件，并执行以下操作：

1. 按修改时间排序。
2. 按四位数字顺序重命名。
3. 文件名中的编号和日期之间使用下划线 `_`。
4. 按界面输入的分组数量切分。
5. 将每组视频移动到新建子文件夹中。

如果最后剩余的视频数量不足一组，也会单独新建一个子文件夹，命名规则与其他子文件夹相同。

## 文件命名规则

格式：

```text
四位编号_当天日期.后缀名
```

示例：

```text
0001_20260310.mp4
0002_20260310.mp4
0030_20260310.mp4
```

说明：

- 编号从 `0001` 开始
- 编号固定四位，不足补零
- 编号和日期之间使用下划线 `_`
- 日期格式固定为 `YYYYMMDD`
- 后缀名保持原扩展名不变

## 子文件夹命名规则

格式：

```text
起始编号-结束编号_当天日期
```

示例：

```text
0001-0030_20260310
0031-0060_20260310
0061-0067_20260310
```

说明：

- 编号区间之间使用短横线 `-`
- 编号区间和日期之间使用下划线 `_`
- 最后一组即使不足设定数量，也会单独创建子文件夹

## 分组数量输入规则

输入框用于指定“每个子文件夹放多少个视频”。

- 只允许输入数字
- 默认值为 `30`
- 允许范围为 `1-1000`
- 点击输入框时会自动全选当前数字
- 不使用鼠标拖选数字
- 关闭窗口时会保存当前输入值
- 下次启动时会恢复上一次保存的值

示例：

- 输入 `30`：每 30 个视频一组
- 输入 `100`：每 100 个视频一组
- 输入 `1`：每个视频单独一个子文件夹

## 配置保存

程序关闭时，会将以下信息保存到 `exe` 所在目录：

- 窗口左上角位置
- 窗口宽高
- 当前分组数量

配置文件名固定为：

```text
VidoWallpaperNumbers.config.json
```

说明：

- 配置文件与 `exe` 放在同一目录
- 如果配置文件不存在，程序会使用默认值启动
- 如果配置文件内容无效，程序会自动回退到安全默认值

## 界面说明

当前界面为原生 Win32 GUI，使用暗色主题，布局为紧凑居中。

界面元素只有三个：

- 数字输入框
- 开始按钮
- 进度条

补充说明：

- 输入框数字居中显示，字号较大
- 文件处理在后台 goroutine 中执行
- 窗口标题栏和界面都使用暗色风格
- 构建后的 `exe` 不弹命令行窗口

## 响应性说明

文件处理在后台执行，UI 线程只负责：

- 处理消息循环
- 更新界面
- 接收进度事件

当前实现的关键策略：

- GUI 消息循环固定在同一 OS 线程，避免 Win32 线程亲和性问题
- 进度更新通过 `PostMessage` 主动唤醒窗口，而不是依赖定时器轮询
- 每次只分批处理有限数量的进度事件，避免单次处理拖住消息循环
- 队列拥堵时允许丢弃部分中间进度事件，但会优先保留完成事件
- 每次开始处理前会清空残留事件，避免上一轮状态污染下一轮
- 后台协程如果出现未处理异常，会回传错误事件，不会让界面永久停在“运行中”

这意味着：

- 处理速度慢时，窗口仍应保持可响应
- 进度条不保证每一步都刷新，但最终状态应能正确到达
- 如果底层磁盘或系统文件调用本身阻塞，处理会变慢，但不应再因为消息机制导致界面假死

## 诊断日志

当前版本默认关闭详细日志输出，不会主动生成日志文件。

说明：

- 诊断日志代码仍然保留在项目中
- 当前打包版默认不初始化日志文件
- 如果后续再次需要排查卡死或异常，可以重新打开日志输出

## 安全性与失败处理

程序使用“两阶段重命名”来降低命名冲突风险：

1. 先把原文件改成临时名
2. 再统一改成最终文件名
3. 最后按分组移动到子文件夹

正式执行前还会做预检查：

- 没有可处理的 `.mp4` 文件时直接报错
- `.mp4` 数量超过 `9999` 时直接报错
- 目标子文件夹已存在时报错
- 目标文件名已被其他文件占用时报错
- 临时文件名已存在时报错

如果处理中途失败，程序会尽量回滚已经处理过的文件和已创建的目录。

## 不主动写入 C 盘

本项目遵守以下约束：

- 优先使用系统环境变量中的 `go`
- 构建缓存、模块缓存、临时目录固定在项目目录下的 `.build`
- 不依赖必须写入 `C:\Users\...` 的构建流程
- 运行配置文件保存到 `exe` 所在目录，不写入注册表、不写入用户目录

当前构建缓存目录：

- `.build/gocache`
- `.build/gomodcache`
- `.build/gotmp`
- `.build/tmp`

## 图标资源

项目根目录中的图片会在构建时自动转换为 `ico`，并嵌入最终生成的 `exe`。

当前规则：

- 从项目根目录查找图片文件
- 支持 `.png`、`.jpg`、`.jpeg`、`.bmp`
- 按文件名排序后取第一张作为图标源文件

## 常见问题

### 为什么只处理第一层 `.mp4`？

这是当前设计。程序只扫描 `exe` 所在目录的第一层文件，不递归处理子目录。

### 文件名和文件夹名为什么格式不同？

这是刻意设计：

- 文件名：`0001_20260310.mp4`
- 文件夹名：`0001-0030_20260310`

文件名强调单个编号，文件夹名强调编号区间。

### 配置文件在哪里？

配置文件保存在 `exe` 同目录，文件名固定为 `VidoWallpaperNumbers.config.json`。

### 如果我删除配置文件会怎样？

程序会按默认值启动：

- 分组数量：`30`
- 窗口尺寸：默认紧凑窗口尺寸
- 窗口位置：交给系统默认放置

### 为什么处理很多文件时进度条不是每一步都跳？

这是当前实现的取舍。为了保证窗口响应，程序会在高频进度更新时丢弃部分中间事件，但最终完成状态和错误状态会优先送达。

## 已知边界

- 当前只处理 `.mp4`
- 当前排序依据是修改时间，不是创建时间
- 当前编号上限是 `9999`
- 当前界面没有“选择目录”功能，工作目录固定为 `exe` 所在目录
